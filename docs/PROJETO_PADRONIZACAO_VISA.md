# Projeto de Padronização do Layout – App VISA Anápolis

**Autor:** Manus AI
**Data:** 28 de Fevereiro de 2026
**Versão:** 1.0

## 1. Introdução

Este documento apresenta um projeto detalhado para a padronização de todas as páginas do aplicativo da Vigilância Sanitária (VISA) de Anápolis. O objetivo é unificar a interface, adotando o novo layout com menu lateral (sidebar) e dashboard, implementado na versão 2.4.3 do arquivo `index.html`. A proposta visa criar uma experiência de usuário coesa e moderna, com a aparência de um aplicativo integrado (SPA - Single Page Application), onde a navegação entre as diferentes seções ocorre de forma fluida, sem a necessidade de recarregar a estrutura principal da página.

## 2. Análise da Arquitetura Atual

A análise do repositório revelou uma arquitetura híbrida, com um layout moderno já aplicado à página principal e um layout mais antigo presente nas demais páginas. O sistema de autenticação é robusto e centralizado, o que viabiliza a implementação da interface unificada.

### 2.1. Layout Padrão (v2.4.3)

O arquivo `index.html` (v2.4.3) introduziu uma estrutura de layout moderna e modular, que servirá como base para todas as outras páginas. Seus principais componentes são:

- **Layout Principal (`<div class="app-layout">`):** Um contêiner flex que organiza a `sidebar` e a área de conteúdo principal.
- **Menu Lateral (`<aside class="sidebar">`):** Barra de navegação fixa à esquerda (em desktops) e retrátil (em dispositivos móveis), contendo links para todas as seções do sistema, informações do usuário e versão.
- **Área de Conteúdo (`<div class="main-area">`):** Espaço principal à direita da `sidebar`, que contém o cabeçalho da página e o conteúdo específico de cada seção.
- **Cabeçalho de Página (`<div class="page-header">`):** Apresenta o título e subtítulo da página atual, localizado dentro da `main-area`.
- **Design System (`design-tokens.css`):** Um sistema de variáveis CSS que garante a consistência de cores, fontes e espaçamentos, incluindo suporte a *dark mode*.

### 2.2. Sistema de Autenticação

Sua principal dúvida sobre a viabilidade de manter a autenticação em um modelo de navegação unificado é pertinente e a resposta é **sim, é totalmente viável**. A análise do fluxo de autenticação confirma isso.

> O sistema utiliza o **Firebase Authentication**, que gerencia o estado do login de forma persistente no navegador (`browserLocalPersistence`). Uma vez que o usuário faz o login no `index.html`, sua sessão permanece ativa. 

As páginas internas que requerem acesso restrito utilizam um módulo JavaScript (`guard.js` ou `js/guard1.js`) que implementa a função `protectPage`. Esta função verifica, no momento do carregamento da página, se existe uma sessão de usuário ativa e autorizada. 

- **Se o usuário está autenticado:** O conteúdo da página é exibido normalmente.
- **Se o usuário não está autenticado:** Ele é automaticamente redirecionado para o `index.html` para realizar o login.

Este modelo é ideal para a arquitetura proposta. O usuário fará login uma única vez e poderá navegar por todas as páginas protegidas através do menu lateral, sem ser interrompido para se autenticar novamente, até que sua sessão expire por tempo ou inatividade.

## 3. Proposta de Padronização

Para alcançar a consistência visual e funcional em todo o aplicativo, propõe-se a refatoração de todas as páginas `.html` para que incorporem a estrutura de layout do `index.html`. O plano consiste em criar um modelo base e aplicá-lo a cada página, movendo o conteúdo específico para dentro do novo contêiner.

### 3.1. Estratégia de Implementação

A migração será feita página por página, seguindo um processo claro para minimizar erros e garantir que a funcionalidade de cada uma seja preservada.

#### Passo 1: Criação do Arquivo de Template (`_template.html`)

O primeiro passo é criar um arquivo `_template.html` no diretório raiz. Este arquivo servirá como o esqueleto para todas as outras páginas. Ele conterá a estrutura completa do `app-layout`, incluindo a `sidebar`, a `main-area`, o `topbar` mobile e o `footer`, deixando a seção de conteúdo principal (`<main class="page-content">`) vazia para ser preenchida.

**Estrutura do `_template.html`:**

```html
<!DOCTYPE html>
<html lang="pt-BR">
<head>
  <!-- Todos os <meta>, <link> para CSS (design-tokens, base, components, etc.) e scripts globais -->
  <title>Vigilância Sanitária</title>
</head>
<body>
  <div id="loading">...</div>
  <div id="login">...</div>

  <div id="conteudo">
    <div class="app-layout">
      <!-- Sidebar (copiada integralmente do index.html) -->
      <aside class="sidebar" id="sidebar">...</aside>

      <!-- Overlay Mobile -->
      <div class="sidebar-overlay" id="sidebarOverlay" onclick="closeSidebar()"></div>

      <!-- Área Principal -->
      <div class="main-area">
        <!-- Topbar Mobile (copiado do index.html) -->
        <header class="topbar">...</header>

        <!-- Conteúdo da Página (a ser preenchido) -->
        <main class="page-content">
          <div class="page-header">
            <h1 class="page-title">TÍTULO DA PÁGINA</h1>
            <p class="page-subtitle">Subtítulo descritivo da página.</p>
          </div>

          <!-- =================================== -->
          <!-- INÍCIO DO CONTEÚDO ESPECÍFICO DA PÁGINA -->
          <!-- =================================== -->

          <!-- Ex: Formulários, tabelas, cards, etc. -->

          <!-- =================================== -->
          <!-- FIM DO CONTEÚDO ESPECÍFICO DA PÁGINA -->
          <!-- =================================== -->
        </main>

        <!-- Footer (opcional, se houver) -->
        <footer class="footer">...</footer>
      </div>
    </div>
  </div>

  <!-- Scripts (Firebase, guard.js, e scripts específicos da página) -->
</body>
</html>
```

#### Passo 2: Refatoração das Páginas

Para cada arquivo `.html` do projeto (ex: `os.html`, `alvara.html`, `cvs.html`, etc.), o seguinte processo será aplicado:

1.  **Substituir Estrutura:** O conteúdo inteiro do arquivo `.html` existente será substituído pelo conteúdo do `_template.html`.
2.  **Mover Conteúdo Específico:** O conteúdo principal da página original (geralmente o que está dentro do `<body>` e fora do `<header>`) será movido para dentro da seção `<main class="page-content">` do novo arquivo.
3.  **Atualizar Cabeçalho:** O título e subtítulo no `<div class="page-header">` serão atualizados para refletir a função da página (ex: "Ordens de Serviço", "Consulta de Alvarás"). O antigo `<header class="app-header">` e o botão `btn-done-ios` serão removidos, pois a navegação agora é centralizada na `sidebar`.
4.  **Ajustar Scripts:** Os scripts `<script>` específicos da página original, especialmente aqueles que manipulam o DOM ou buscam dados, serão movidos para o final do `<body>` no novo arquivo. O script de proteção `protectPage` (ou `onAuthStateChanged`) deve ser mantido para as páginas que exigem autenticação.
5.  **Ativar Link no Menu:** No código da `<aside class="sidebar">`, a classe `active` será movida do link do `Dashboard` para o link correspondente à página que está sendo editada. Por exemplo, ao editar `os.html`, o item de menu "OS" receberá a classe `active`.

#### Passo 3: Centralização e Limpeza

- **CSS:** Os estilos que foram duplicados em vários arquivos (como os do `app-header` antigo) podem ser removidos, e os estilos específicos de cada página devem ser mantidos ou movidos para arquivos CSS separados para melhor organização.
- **JavaScript:** Funções JavaScript duplicadas, como as de manipulação de modais ou loading, podem ser centralizadas em um arquivo `app.js` global, a ser carregado em todas as páginas.

## 3. Arquitetura da Solução

Para alcançar a padronização desejada, proponho uma arquitetura baseada em componentes reutilizáveis e uma estrutura de arquivos clara e organizada. A ideia central é que cada página seja construída a partir de um layout base, herdando o menu lateral, o cabeçalho e o rodapé, e apenas o conteúdo principal seja específico de cada página.

### 3.1. Estrutura de Arquivos e CSS

A estrutura de CSS será modularizada para facilitar a manutenção e o reuso. Os arquivos CSS existentes (`design-tokens.css`, `header-component.css`, etc.) serão a base, e criaremos novos arquivos para componentes específicos, como o menu lateral e o layout principal.

```
/css/
  ├── design-tokens.css       # Variáveis globais (cores, fontes, etc.)
  ├── main-layout.css         # Estilos do layout principal (grid, flexbox)
  ├── sidebar-component.css   # Estilos do menu lateral
  ├── header-component.css      # Estilos do cabeçalho
  └── ...                     # Outros componentes
```

### 3.2. Componentes Reutilizáveis (HTML/JS)

Para evitar a duplicação de código, criaremos "componentes" HTML que serão carregados dinamicamente em todas as páginas via JavaScript. Isso garante que, ao alterar o menu lateral, por exemplo, a modificação seja refletida em todas as páginas que o utilizam.

*   **`sidebar.html`**: Conterá apenas o HTML do menu lateral.
*   **`header.html`**: Conterá o HTML do cabeçalho do dashboard.

Um script JavaScript (`component-loader.js`) será responsável por buscar o conteúdo desses arquivos e inseri-los no local correto em cada página.

### 3.3. Layout Padrão (`main-layout.html`)

Criaremos um arquivo `main-layout.html` que servirá como um template para todas as outras páginas. Ele conterá a estrutura básica do layout, com os *placeholders* (locais) onde o menu, o cabeçalho e o conteúdo específico da página serão inseridos.

```html
<!DOCTYPE html>
<html lang="pt-br">
<head>
    <!-- Metatags, links de CSS, etc. -->
    <link rel="stylesheet" href="css/main-layout.css">
    <link rel="stylesheet" href="css/sidebar-component.css">
    <link rel="stylesheet" href="css/header-component.css">
</head>
<body>
    <div class="app-container">
        <aside id="sidebar-container"></aside>
        <main class="main-content">
            <header id="header-container"></header>
            <div id="page-content">
                <!-- O conteúdo específico de cada página será carregado aqui -->
            </div>
        </main>
    </div>
    <script src="js/component-loader.js"></script>
</body>
</html>
```

## 5. A Questão da Autenticação

Sua dúvida sobre a autenticação é pertinente e central para o sucesso do projeto. A análise revelou que o sistema de autenticação, embora implementado de formas ligeiramente diferentes entre as páginas, pode ser unificado e integrado ao novo layout sem grandes problemas.

### 5.1. Como a Autenticação Funcionará no Novo Modelo

O fluxo de autenticação permanecerá conceitualmente o mesmo, mas será mais robusto e centralizado:

1.  **Ponto de Entrada (`index.html`):** O usuário continuará a fazer o login através do `index.html`. O `onAuthStateChanged` continuará a ser o principal observador do estado de autenticação.
2.  **Guarda de Página (`guard.js`):** O script `guard.js` (ou uma versão unificada dele) será incluído em **todas** as páginas que requerem autenticação, mas **após** o carregamento do layout principal. A lógica será a seguinte:
    *   A página tenta carregar.
    *   O `component-loader.js` insere o menu e o cabeçalho.
    *   O `guard.js` é executado. Ele verifica o estado de autenticação do Firebase (que é persistente no navegador).
    *   **Se o usuário não estiver logado**, o `guard.js` redireciona para o `index.html` **antes** que qualquer conteúdo sensível seja exibido.
    *   **Se o usuário estiver logado**, a execução da página continua normalmente, e o conteúdo principal é exibido.

Isso resolve a questão: a navegação entre as páginas através do menu lateral funcionará perfeitamente. Se um usuário não autenticado tentar acessar uma página protegida diretamente pela URL, o `guard.js` o interceptará e o enviará para a página de login. O usuário autenticado, por sua vez, navegará de forma fluida, pois seu estado de login é verificado em cada carregamento de página.

### 5.2. Unificação dos Scripts de Guarda

Os scripts `guard.js` e `guard1.js` serão unificados em um único `auth-guard.js` mais genérico, que poderá ser configurado para diferentes níveis de permissão, se necessário no futuro.
